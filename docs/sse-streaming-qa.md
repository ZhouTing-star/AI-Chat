# SSE 流式问答（本项目）

以下内容结合本项目的前后端实现，按问题给出“专业版解答”和“面试版回答”。

---

## 1. SSE 的原理？为什么用 SSE 实现流式回答，有了解过其他方案吗

**专业版解答**
- SSE（Server-Sent Events）是基于 HTTP 的单向推送机制，浏览器通过持久连接接收服务器持续发送的文本流。服务端按 SSE 协议输出：`event:`、`data:` 行，并以空行分隔消息。
- 本项目使用 SSE 的原因：
  - 模型输出天然是“单向流”，客户端只需要接收增量，不需要双向实时交互；SSE 简化实现。
  - 兼容性好，HTTP/1.1 长连接即可，部署成本低。
  - 调试友好，服务端可直接通过 `res.write()` 输出事件，前端用 `fetch + ReadableStream` 逐块解析。
- 其他方案：
  - WebSocket：双向通信，适合需要频繁双向消息或多人协作的场景，但复杂度更高。
  - HTTP Chunked（纯文本流）：实现简单，但缺少事件语义，难以区分 `done` / `citations` 等事件。
  - gRPC / WebTransport：性能更高但部署成本和生态复杂度更高。
- 结合本项目：前端是 `fetch` 的流式读取，后端把上游 LLM 的 SSE 转发为自己的 SSE 格式，便于统一处理 `citations` 和 `done` 事件。

**面试版回答**
- SSE 是基于 HTTP 的单向推送，用一个长连接不断发 `data:` 事件给浏览器。
- 本项目只需要“模型往前端推”，不需要双向，所以用 SSE 成本低、实现快、部署简单。
- 了解 WebSocket/gRPC，但这类需要双向或更复杂协议时再选，本项目 SSE 最合适。

---

## 2. 详细说一下整个流式对话的过程具体是怎么实现的；打字机效果是怎么实现的

**专业版解答**
- 前端流程：
  1. 发送消息时，先在本地插入一条空的 assistant 消息，状态为 `streaming`。
  2. 调用 `streamChatReply()` 通过 `fetch` 发起 POST，并开启 `ReadableStream` 读取 SSE。
  3. 前端解析 `event:` / `data:` 行并增量拼接到消息内容；`citations` 事件更新引用；`done` 事件结束流。
  4. 结束后将状态设为 `done`，并更新会话预览。
- 后端流程：
  1. `/api/chat/stream` 接口读取入参，进行 RAG 检索并可先发送 `citations` 事件。
  2. 使用 `axios` 以流式方式请求上游模型（`stream: true`）。
  3. 使用 `eventsource-parser` 解析上游 SSE，抽取 delta 后转写给前端 SSE。
  4. 发送 `done` 事件并关闭连接。
- 打字机效果：
  - 本项目的“打字机”本质是 **模型流式增量** 带来的逐段更新，并没有额外的定时器逐字渲染。
  - UI 仅在收到 chunk 时追加到消息内容，配合“正在生成”的三点脉冲指示器。

**面试版回答**
- 前端先创建空的 AI 消息，收到 chunk 就拼接内容，`done` 就结束；后端把模型的流式结果解析后再转成 SSE 推给前端。
- 打字机效果是流式增量天然形成的，不是人为用定时器逐字显示。

---

## 3. chunk 增量渲染会导致频繁重渲染吗？项目中你是怎么做的

**专业版解答**
- 确实会有更频繁的状态更新，但本项目做了三层控制：
  1. **虚拟列表**：消息列表用 `react-window`，只渲染可视区域，避免全量重绘。
  2. **单条消息 memo**：`MessageItem` 使用 `memo`，未变更的消息不会重复渲染。
  3. **滚动防抖**：自动滚动用 `debounce` + `requestAnimationFrame`，避免频繁滚动触发布局抖动。
- 结果是：每个 chunk 只更新末尾一条 assistant 消息，且渲染范围被控制在可视区内，整体性能可控。

**面试版回答**
- 会更频繁，但我们用虚拟列表 + 单条消息 memo，只有最后一条在动。
- 自动滚动也做了防抖，所以整体不卡。

---

## 4. 怎么实现流式回答的暂停、继续、重新生成

**专业版解答**
- 暂停：前端保存每个会话的 `close()`，用户点“暂停”时调用 `AbortController` 中断 fetch，标记会话为 `paused`。
- 继续：保存上一次流式任务的元信息（prompt、模型、上下文、assistantId）。恢复时把已输出内容作为已完成消息，拼接“继续回答”的提示词，再次发起流式请求并复用同一条 assistant 消息继续追加。
- 重新生成：先中断当前流，再删除上一条 assistant 消息，用原始 prompt 和上下文重新请求。

**面试版回答**
- 暂停就是 `abort` 连接；继续用之前的 prompt 和已输出内容拼一个“从这里继续”的提示词再开新流；
- 重新生成就是删掉旧回答，再用原 prompt 重新请求一次。

---

## 5. SSE 异常怎么处理？断网、超时、服务崩溃怎么兼容

**专业版解答**
- 前端：
  - `fetch` 非 2xx 状态会读取错误信息并触发 `onError`。
  - 响应体为空或读流异常会提示“连接异常”。
  - 主动中断（暂停）不会提示错误。
- 后端：
  - SSE 响应加心跳注释 `: keepalive`，避免代理空闲断开。
  - 上游流读取失败、上游异常状态码会转为 SSE `error` 事件返回给前端。
  - 进程层面有 `uncaughtException` / `unhandledRejection` 兜底日志，避免无日志崩溃。
- 断网/服务崩溃：前端会收到流读异常并结束，用户可点击“继续/重新生成”进行恢复。
- 超时：本项目默认不设置上游超时（`timeout: 0`），避免长回答被切断；如果要更稳健，可以加客户端超时和重试策略。

**面试版回答**
- 前端对非 2xx、空响应、读流异常统一提示；后端上游异常也会转成 `error` 事件。
- 断网/崩溃会触发错误提示，用户可继续或重新生成。

---

## 6. SSE 可以携带自定义请求头吗？项目中有处理鉴权吗

**专业版解答**
- 可以。SSE 如果用 `EventSource` 会限制自定义请求头，但本项目用的是 `fetch` 流式读取，因此可以自由设置请求头。
- 项目中支持可配置的鉴权头：
  - 前端读取 `VITE_API_TOKEN_HEADER` 和 `VITE_API_TOKEN_PREFIX`，把 `localStorage` 中的 token 作为请求头发送。
  - 这样可以适配不同后端规范（`Authorization: Bearer xxx` 或自定义头）。

**面试版回答**
- 可以。我们不是 `EventSource`，而是 `fetch` 流式请求，所以自定义请求头没问题。
- 项目里有可配置的 token header/prefix 做鉴权。

---

## 7. 同一浏览器，复制一个标签页，在这两个标签页中分别提问会怎么样

**专业版解答**
- 当前实现是“每个标签页一套内存状态”，`zustand` 的持久化只写入 localStorage，但**没有自动跨标签同步**。
- 结果：
  - 两个标签页会各自发起 SSE 请求，不会互相中断。
  - 如果使用同一个会话 ID，最后一次写入 localStorage 的内容可能覆盖另一个标签页的持久化状态。
  - UI 可能出现短暂不一致（每个标签页渲染的是自己内存中的消息列表）。
- 后端不依赖 sessionId 做会话锁或并发限制，因此并发问答会并行进行。
- 如果需要更严格的一致性，可以监听 `storage` 事件或用 BroadcastChannel 做跨标签同步。

**面试版回答**
- 两个标签页各自独立发 SSE，不会互相影响。
- 但 localStorage 没做跨标签同步，可能出现覆盖或显示不一致。
- 后端不限制并发，所以两个问题会并行处理。

# RAG 检索问答（本项目）

以下内容结合本项目的后端实现（SQLite + FTS5 + 向量检索），按问题给出“专业版解答”和“面试版回答”。

---

## 1. 什么是 RAG？你的项目里 RAG 流程是什么？

**专业版解答**
- RAG（Retrieval-Augmented Generation）是“先检索、再生成”的模式，通过从知识库检索相关文档片段，增强模型回答的准确性与可控性。
- 本项目的 RAG 流程：
  - 文档上传后解析文本、分块、向量化写入 SQLite。
  - 提问时后端先进行检索（向量或混合），拿到 topK 片段。
  - 将检索结果拼接为系统提示词插入对话上下文，再请求上游模型流式生成。
  - 同时把引用片段通过 SSE `citations` 事件返回前端做透明展示。

**面试版回答**
- RAG 就是“先检索再生成”，用资料增强模型回答。
- 我们项目里先检索文档片段，再把结果塞进系统提示词，让模型基于资料回答，并把引用返回前端展示。

---

## 2. 你用了 FTS5 + 向量混合检索，为什么这么做？

**专业版解答**
- 纯向量检索对语义相似更好，但对关键词、专有名词、代码片段的精确命中不稳定。
- FTS5 属于高效的全文检索，能保证关键词和短语的精确匹配。
- 混合检索可以兼顾“语义召回 + 关键词召回”，提升召回率和稳定性。
- 本项目支持 `hybrid`/`vector`/`off` 等模式，在准确性与速度之间可配置。

**面试版回答**
- 向量强在语义，FTS5 强在关键词；混合能避免漏掉专有名词或短语。
- 项目提供多种检索模式，默认用混合模式更稳。

---

## 3. 文档解析、分块怎么做的？怎么处理大文件？

**专业版解答**
- 解析：
  - PDF 用 `pdf-parse` 解析文本；DOCX 用 `mammoth`；文本类直接 UTF-8 解码。
  - 解析为空或失败时给出降级文本说明，保证入库可追溯。
- 分块：
  - 先按句子边界切分，再合并到目标长度（默认 420 字），设置 80 字符重叠，保证语义连续。
- 大文件：
  - 上传层限制单文件大小（默认 30MB）。
  - 解析与分块均在服务端进行，分块后向量化写入，降低单次检索负担。

**面试版回答**
- PDF/DOCX/文本分别用对应工具解析，解析失败给降级文本。
- 分块按句子切，再合并到 400+ 字并做 80 字重叠。
- 大文件有上传大小限制，分块后再入库。

---

## 4. 什么是 FTS5？它在项目里作用是什么？

**专业版解答**
- FTS5 是 SQLite 的全文检索扩展，支持倒排索引与快速文本匹配。
- 本项目用 FTS5 为文档块建立全文索引，在混合检索中提供关键词召回通道。
- 结合向量检索后可提升召回稳定性，尤其对专有名词、数字、代码片段更有效。

**面试版回答**
- FTS5 是 SQLite 的全文检索引擎，用来做关键词匹配。
- 项目里用于和向量检索配合，提高命中率。

---

## 5. 动态阈值策略是什么？为什么要用？

**专业版解答**
- 项目中通过 `strict / balanced / general` 方式控制检索模式与命中阈值：
  - `strict` 更高阈值（约 0.7），命中不足直接拒答。
  - `balanced` 中等阈值（约 0.4），既保证相关性又避免召回过少。
  - `general` 直接跳过检索，使用通用知识回答。
- 目的：不同业务场景对“准确性 vs 覆盖率”权衡不同，动态阈值可以显式控制风险。

**面试版回答**
- 严格/平衡/通用三种模式，相当于不同阈值策略。
- 严格模式命中不够就拒答，平衡模式更稳，通用模式直接跳过检索。

---

## 6. 引用透明展示怎么实现？用户怎么知道答案来自哪段文档？

**专业版解答**
- 后端检索后把命中的文档片段作为 `citations` 事件通过 SSE 发送到前端。
- 前端将引用按分数排序，截取预览文本，并在消息底部展示“透明度徽章”。
- 用户可以展开引用列表查看具体来源、相似度分数与片段内容，从而判断答案依据。

**面试版回答**
- 后端把引用片段通过 SSE 事件发给前端。
- 前端展示引用卡片和相似度，让用户知道答案来自哪里。

---

## 7. 怎么解决知识库检索不准确的问题？

**专业版解答**
- 分块策略：句子边界 + 重叠，避免语义断裂。
- 检索模式：可切换 `hybrid` / `vector` / `strict`，根据场景调优。
- 上下文控制：限制上下文长度，避免“噪声”稀释检索结果。
- 质量提示：严格模式下如果没有高质量命中直接拒答，避免模型胡答。
- 若仍不准，可增加文档质量、做 OCR、优化分块长度或增加关键词字段。

**面试版回答**
- 先保证分块和重叠合理，减少语义丢失。
- 用混合检索或严格模式提升准确率，命中不够就拒答。
- 文档质量和分块长度也要调优。

---

## 8. 多文件上传、并发处理怎么做？

**专业版解答**
- 前端支持多文件选择，逐个解析后入队；上传状态（queued/uploading/done/failed）独立管理。
- 后端使用 `multer` 接收文件并在内存解析，逐文件写入 SQLite，避免互相阻塞。
- 前端使用 `Promise.allSettled` 并行处理文件解析，保证单个失败不影响整体。

**面试版回答**
- 前端可多选文件并行处理，状态独立。
- 后端逐文件解析入库，失败不会影响其他文件。

---

# 消息渲染与引用透明度（本项目）

以下内容结合前端实现，说明 AI 消息富文本渲染、代码高亮、Top-K 引用展开与低质量引用折叠的做法与技术选型。

---

## 1. AI 消息富文本渲染（支持 Markdown）是怎么实现的？用了什么技术？

**专业版解答**
- 使用 `react-markdown` 解析 Markdown，并通过 `remark-gfm` 支持表格、任务列表等 GFM 语法。
- 对标题、段落、列表、链接、表格等元素做了组件级样式映射，保持视觉一致性。
- 对内联代码与代码块做了区分：有语言标识的渲染为高亮代码块，否则渲染为内联 code。

**面试版回答**
- 用 `react-markdown` + `remark-gfm` 做 Markdown 渲染。
- 自定义组件映射来控制标题、列表、链接和表格样式。

---

## 2. 代码高亮怎么实现？

**专业版解答**
- 使用 `react-syntax-highlighter` 的 Prism 方案，并选用 `oneDark` 主题。
- 通过 `language-xxx` 类名解析语言标识，渲染为高亮代码块。
- 同时提供“复制代码”按钮，提升可用性。

**面试版回答**
- 用 `react-syntax-highlighter`（Prism）做高亮，主题用 `oneDark`。
- 识别 `language-xxx` 决定高亮语言，并提供复制按钮。

---

## 3. 引用透明度展示（Top-K 展开 + 低质量折叠）怎么实现？

**专业版解答**
- 后端通过 SSE `citations` 事件下发引用片段与分数；前端在消息组件中进行排序、清洗与截取。
- 只展示 Top-K（默认 5 条），并按分数划分“高质量引用”和“低质量引用”。
- 高质量引用默认展示 1 条，可展开“查看全部高质量”；低质量引用默认折叠，点击后展开。
- 透明度徽章根据模式与最高分数显示“基于资料/部分相关/AI 通用知识”等提示。

**面试版回答**
- 引用来自后端 SSE 事件，前端按分数排序，默认只展示 Top-K。
- 高质量可展开、低质量折叠，徽章提示引用可靠度。

---

## 4. 这些功能对应的关键实现位置在哪里？

**专业版解答**
- 富文本与代码高亮：在消息渲染组件中完成（`react-markdown` + `react-syntax-highlighter`）。
- 引用透明度与折叠逻辑：在消息组件内通过分数阈值和状态控制实现。
- 引用事件来源：来自 SSE 流式返回的 `citations` 事件。

**面试版回答**
- 消息渲染与引用展示都在单条消息组件完成；引用数据来自 SSE 事件。

# 状态管理（Zustand）问答（本项目）

以下内容结合本项目的状态管理实践，按问题给出“专业版解答”和“面试版回答”。

---

## 1. 为什么用 Zustand 进行状态管理，了解 Redux 吗？

**专业版解答**
- Zustand 轻量、API 简洁，不需要大量样板代码，适合中小型项目快速落地。
- 支持按需订阅与细粒度更新，配合 React hooks 使用成本低。
- 了解 Redux：它生态成熟、规范化强，但在本项目规模下会引入较多样板与配置成本。

**面试版回答**
- Zustand 轻量、上手快，适合这个项目体量。
- Redux 我了解，但这里用它会偏重，成本不划算。

---

## 2. 为什么拆分成 会话 / 聊天 / UI 三个 Store？项目状态管理思路是怎样的？

**专业版解答**
- 按“生命周期与职责”拆分：
  - 会话 Store：会话列表与当前会话元信息。
  - 聊天 Store：消息内容、流式状态、暂停等核心业务数据。
  - UI Store：输入框、页面切换、上传进度等界面态。
- 这样可以降低耦合、避免单 Store 过大，便于分模块维护与测试。

**面试版回答**
- 会话、消息、UI 的生命周期不一样，所以拆成三个 Store。
- 拆分后更清晰，避免一个 Store 过重。

---

## 3. Zustand 持久化怎么实现？

**专业版解答**
- 使用 `persist` 中间件，把需要的状态写入 localStorage。
- 会话与消息做持久化，流式状态、暂停状态等临时状态不持久化。
- 对 `Map` 类型做了自定义序列化/反序列化，保证可恢复。

**面试版回答**
- 用 `persist` 中间件把状态存到 localStorage。
- 只持久化必要的数据，临时状态不存。

---

## 4. 什么是不可变更新？Zustand 里怎么实现？

**专业版解答**
- 不可变更新是指每次修改都返回“新对象/新引用”，避免原地修改。
- 在 Zustand 中通过 `set((state) => ({ ... }))` 创建新对象，必要时复制数组或 Map，再写回。
- 本项目在更新消息时都创建新的数组或新的 Map，保证 React 能正确触发渲染。

**面试版回答**
- 不可变更新就是不直接改原对象，而是生成新引用。
- Zustand 里通过 `set` 返回新对象，数组/Map 都会先复制再改。

---

## 5. 多个组件共享状态，Zustand 怎么保证不重复渲染？

**专业版解答**
- Zustand 支持按需订阅，组件只订阅自己关心的切片。
- 状态更新时，只有订阅了变化切片的组件会重新渲染。
- 本项目中消息列表、顶栏、输入区分别订阅不同状态，避免整体重绘。

**面试版回答**
- Zustand 是按切片订阅的，组件只会在自己关心的字段变化时重渲染。
- 所以共享状态也不会导致全局重复渲染。

# 长列表性能与虚拟滚动（本项目）

以下内容结合 `react-window` 的实现，按问题给出“专业版解答”和“面试版回答”。

---

## 1. react-window 原理，项目中具体是怎么实现虚拟滚动的？

**专业版解答**
- 原理：只渲染可视区域内的列表项，通过“滚动容器 + 计算可视范围”动态挂载/卸载行，减少 DOM 数量。
- 项目中用 `List` 组件作为虚拟列表容器：
  - 传入 `rowCount`、`rowHeight`、`rowComponent`。
  - `rowHeight` 支持动态高度，未测量时用估算值。
  - `overscanCount` 预渲染部分行，降低滚动抖动。

**面试版回答**
- react-window 只渲染可视区的行，滚动时动态替换。
- 项目里用 `List` + 行高函数实现虚拟滚动，减少 DOM。

---

## 2. 你实现了动态高度、防抖滚动、组件缓存，分别讲一下？

**专业版解答**
- 动态高度：每条消息用 `ResizeObserver` 监听高度变化，测量后写入 `rowSizes`，虚拟列表用该高度精确计算。
- 防抖滚动：自动滚动到底部时做了 `debounce` + `requestAnimationFrame`，避免频繁滚动触发布局重排。
- 组件缓存：单条消息组件用 `memo` 包裹，未变化的消息不会重复渲染。

**面试版回答**
- 动态高度用 `ResizeObserver` 实时测量。
- 自动滚动加防抖，减少抖动。
- 消息组件 `memo` 缓存，避免重复渲染。

---

## 3. 长对话列表性能优化你还做了哪些？

**专业版解答**
- 只渲染可视区（虚拟列表）。
- 单条消息组件 `memo`，减少不必要更新。
- 自动滚动使用防抖，避免频繁 reflow。
- 消息高度变化只更新对应行，不触发全量列表更新。

**面试版回答**
- 核心是虚拟列表 + `memo`。
- 自动滚动加防抖，减少渲染压力。

---

## 4. 自动滚动过程中用户想往上滑查看历史怎么办？

**专业版解答**
- 监听滚动事件：用户向上滚动时暂停自动跟随（`shouldFollowRef = false`）。
- 当用户回到底部附近（距离小于阈值）时再恢复自动跟随。
- 这样既保证新消息能跟随，又不会打断用户查看历史。

**面试版回答**
- 用户往上滚动就停止自动滚动；
- 回到底部附近再自动跟随。
