# AI Chat Platform 项目梳理

## 1. 项目概述

AI Chat Platform 是一个前后端分离的智能问答 MVP：

- 前端：提供多会话聊天、流式回复、知识库管理与检索测试界面。
- 后端：负责模型流式转发（SSE）、文档上传解析、向量化入库与 RAG 检索。
- 数据层：使用本地 SQLite 持久化知识库、文档、切片与向量。

项目核心目标是把“通用聊天能力”和“本地知识库增强回答能力”打通，形成完整链路：

1. 用户输入问题。
2. 后端按策略从知识库检索候选切片。
3. 检索结果拼接成系统提示注入上下文。
4. 请求上游 LLM 并以 SSE 实时回流到前端。
5. 前端展示答案及引用来源，支持暂停/继续/重生。

## 2. 所用技术栈

### 2.1 前端技术栈

- React 19 + TypeScript + Vite
  - 单页应用开发、类型约束与快速构建。
- Zustand（含 persist）
  - 管理会话、消息、UI、主题、用户权限状态，并落地本地持久化。
- react-window
  - 聊天消息虚拟列表，降低长对话渲染压力。
- react-markdown + remark-gfm + react-syntax-highlighter
  - 渲染 Markdown、GFM（表格/列表）、代码高亮与复制。
- Tailwind CSS 4
  - 快速构建响应式界面与状态样式。

### 2.2 后端技术栈

- Node.js + Express 5
  - 提供 REST + SSE API。
- axios + eventsource-parser
  - 调用上游模型接口并解析流式事件。
- multer
  - 处理文档上传（内存模式 + 体积限制）。
- mammoth + pdf-parse
  - 解析 DOCX/PDF 文档正文。
- better-sqlite3
  - 本地 SQLite 高性能读写，支持 FTS 与事务。

### 2.3 检索与数据技术

- SQLite + FTS5
  - 提供关键词检索能力（BM25 排序）。
- 向量检索（Embedding）
  - 通过 embedding 接口生成向量，使用余弦相似度排序。
- 混合检索（Hybrid）
  - 融合向量分数与关键词分数，提升召回质量和鲁棒性。

### 2.4 工程与开发配置

- ESLint + TypeScript ESLint
  - 统一代码规范与静态检查。
- 前后端分离脚本
  - 根目录脚本用于前端开发构建，server 子项目脚本用于后端运行。

## 3. 主要实现功能及如何实现的

### 3.1 多会话聊天

能力：

- 新建会话、切换会话、会话预览更新。
- 各会话消息独立存储，互不干扰。

实现方式：

- `sessionStore` 维护会话元数据（标题、预览、模型、更新时间）。
- `chatStore` 以 `Map<sessionId, ChatMessage[]>` 维护消息。
- 通过 `persist` 把会话与消息写入 localStorage，刷新后可恢复。

### 3.2 SSE 流式回复

能力：

- 用户发送后，助手内容按 chunk 实时追加。
- 识别 `done` 结束事件、错误事件、引用事件。

实现方式：

- 前端 `streamChatReply` 通过 `fetch + ReadableStream` 按 SSE 协议解析 `event/data`。
- 后端 `/api/chat/stream` 调用上游模型流式接口，并二次转发为标准 SSE。
- 前端把 chunk 增量写入当前 assistant 消息，直至结束。

### 3.3 暂停/继续/重新生成

能力：

- 流式过程中可暂停。
- 暂停后可继续生成，尽量避免重复。
- 支持基于上一任务“重新生成”。

实现方式：

- 每个会话维护单独流连接关闭器（close 函数）与任务元数据。
- 暂停：主动关闭 SSE 连接并标记消息为 paused。
- 继续：构造“续写提示 + 既有上下文”重新请求同一任务。
- 重新生成：删除旧 assistant 消息，复用原 prompt/context 发起新流。

### 3.4 聊天消息渲染与可读性

能力：

- Markdown 渲染、代码高亮、代码复制。
- 消息引用展示（来源、分数、摘要）。

实现方式：

- `MessageItem` 通过 Markdown 组件管线渲染文本。
- 对代码块启用 Prism 高亮并提供复制按钮。
- 将 RAG 返回的 citations 做过滤、排序、截断并可展开查看。

### 3.5 大规模消息列表性能优化

能力：

- 长会话下保持滚动和渲染流畅。

实现方式：

- `MessageList` 使用 `react-window` 虚拟化。
- 结合 `ResizeObserver` 动态测量行高。
- 跟随滚动采用“接近底部自动跟随”策略，避免打断用户阅读历史消息。

### 3.6 知识库管理

能力：

- 查看知识库与文档列表。
- 上传文档、删除文档、启停知识库/文档。
- 触发重建索引。
- 对检索效果进行测试（hybrid/vector/off）。

实现方式：

- 前端 `KnowledgeBasePage` + `knowledgeApi` 对接后端 REST API。
- 后端提供 `/api/rag/...` 一组管理接口。
- 上传采用 `XMLHttpRequest` 以支持进度回调。

### 3.7 文档解析、切片与入库

能力：

- 支持 txt/md/csv/json/pdf/docx。
- 自动切片并写入数据库与检索索引。

实现方式：

- 上传文件先按类型解析正文（PDF、DOCX 有专门解析器）。
- 文本经分句与重叠窗口策略切片（chunk + overlap）。
- 每个切片写入 `chunks` 表与 `chunk_fts` FTS 索引。
- 同步尝试生成向量，失败时允许降级入库（后续可重建补齐）。

### 3.8 RAG 检索增强回答

能力：

- 支持向量检索、混合检索、关闭检索。
- 将检索结果注入系统提示，并随回答返回可追溯引用。

实现方式：

- `ragStore.search` 先筛选有效知识库与文档。
- 向量检索：query 向量 vs chunk 向量，余弦相似度排序。
- 关键词检索：FTS5 + BM25 得分。
- 混合模式按权重融合两类分数并返回 topK。
- 后端把检索结果拼装为 RAG system prompt 注入 LLM messages。

### 3.9 上下文与鲁棒性控制

能力：

- 限制上下文长度，避免请求过大。
- 处理上游异常、网络中断、空响应等问题。

实现方式：

- 后端按“最大条数/单条长度/总长度”三级裁剪消息上下文。
- SSE 保活心跳、异常捕获、统一错误结构回传。
- 前端对错误进行消息级展示，避免整页中断。

## 4. 技术亮点

### 4.1 前后端流式链路闭环完整

不仅实现前端流式渲染，还处理了连接生命周期（建立、结束、中断、恢复）与事件规范（chunk/citations/done/error），具备较好的工程可用性。

### 4.2 混合检索策略实用

向量检索与关键词检索融合，兼顾语义召回与关键词精确匹配，优于单一检索策略，适合中文业务文档场景。

### 4.3 文档入库具备容错与降级能力

即使 embedding 失败也可先完成文档入库，后续通过重建补齐向量，避免“全或无”的导入失败体验。

### 4.4 数据建模与持久化设计清晰

前端状态按会话/消息/UI/权限分仓管理，后端以 knowledge_bases/documents/chunks/fts 分层建模，职责边界明确、扩展路径清楚。

### 4.5 性能优化做在关键路径

聊天区使用虚拟列表 + 动态行高测量，显著降低大量消息时的 DOM 成本；同时具备滚动跟随策略，兼顾性能与交互体验。

### 4.6 协议兼容与可配置性较强

LLM 与 embedding 的 URL、路径、Header、前缀都可通过环境变量配置，兼容不同厂商接口形态，便于替换模型服务。

### 4.7 可追溯回答体验

回答展示引用来源和得分，能帮助用户判断答案依据，提升可解释性与可信度。

## 补充说明

当前项目定位为 MVP：

- 已具备聊天 + 本地知识库 + RAG 的核心能力闭环。
- 鉴权、任务队列、生产级监控与多租户隔离仍可作为后续演进方向。
